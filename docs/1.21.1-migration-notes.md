# GTMM 1.21.1 Migration Notes

本文件记录 1.21.1 移植中的临时适配点。部分差异来自 GTM 1.21.1 尚未完全同步 GTM 1.20.1 主分支 API，因此当前以“先保证 1.21.1 可编译/可运行，后续随 GTM 同步复核语义”为原则处理。

## 临时适配

- `IDropSaveMachine`：GTM 1.21.1 当前没有该接口，`AdvancedBlockPattern` 和 `AdvancedBlockNoAEPattern` 中的拆除保存逻辑临时改为 `MetaMachine.saveToItem(dropStack, registryAccess)`，对齐 GTM 1.21.1 `MetaMachineBlock#getDrops` 的保存路径。后续如果 GTM 1.21.1 同步回 `IDropSaveMachine#saveBreak()` 语义，需要复核自动拆除是否仍要尊重 `saveBreak()` 条件。

- `GTCEuAPI.RegisterEvent`：GTM 1.21.1 已改用 NeoForge `RegisterEvent` 注册流程，GTMM 当前用 `RegisterEvent#getRegistryKey()` 映射 1.20.1 的分阶段顺序：`Registries.CREATIVE_MODE_TAB` 对应创造物品栏阶段，`Registries.BLOCK` / `Registries.ITEM` 对应 GTMM 方块和物品阶段，`GTRegistries.RECIPE_TYPE_REGISTRY` 保留空的配方类型阶段，`GTRegistries.MACHINE_REGISTRY` 注册多方块与机器，`GTRegistries.COVER_REGISTRY` 注册覆盖板。
- `RegisterEvent` 优先级：GTM 1.21.1 的 `GTRegistries.actuallyRegister` 和 `GTRegistrate#onRegister` 都在 `EventPriority.LOW`，GTMM 使用默认 `addListener` 的 `NORMAL` 优先级注册机器/覆盖板阶段，确保自定义 `GTRegistries.register(...)` 发生在 GTM/Registrate 的低优先级消费阶段之前。

- `FluidIngredient` / `SizedIngredient`：GTM 1.21.1 的配方能力已使用 NeoForge `SizedFluidIngredient` 与 `SizedIngredient`。当前先按 1.21.1 类型迁移，后续同步 GTM 主分支时需检查 GTM 是否追加了兼容包装或扩展方法。

- `ItemStackTransfer`：LDLib 1.21.1 当前移除了旧 `com.lowdragmc.lowdraglib.misc.ItemStackTransfer`。当前仅在创造输入总线的 phantom storage 中改用 GTCEu `CustomItemStackHandler`，后续如 LDLib/GTM 提供新等价容器，再评估是否替换。
- `ComponentItem::create`：GTM 1.21.1 的 `ComponentItem` 当前只保留 `ComponentItem(Properties)` 构造器，GTMM 物品注册先改为 `ComponentItem::new`。如果后续 GTM 重新提供 addon 侧工厂方法，可复核是否回到统一工厂。
- `ItemStack` 自定义数据：MC/NeoForge 1.21.1 已迁移到 data component。GTMM 目前用 `ItemStackNbtUtils` 读写 `DataComponents.CUSTOM_DATA`，创造流体单元主路径改为 GTM `GTDataComponents.FLUID_CONTENT`，同时保留旧 `CUSTOM_DATA/Fluid` 的读取兜底，便于旧存档数据过渡。
- `Component.translatable` 参数：MC 1.21.1 的 `TranslatableContents` 会校验参数类型，只允许 `Component`、`Number`、`Boolean`、`String`。无线监控/终端/Jade 提示中直接传入 `ResourceLocation` 或 `UUID` 会在打开 UI/显示 tooltip 时抛 `IllegalArgumentException`，当前已将维度 ID 与 UUID 显式转为字符串。
- LDLib 物品 UI 双端构造：`HeldItemUIFactory` 在服务端打开 UI 后，客户端处理 `ldlib:ui_open` 时也会调用一次物品组件的 `createUI`。因此物品 UI 构造逻辑不能假设 `Player#level()` 一定是 `ServerLevel`。无线能源终端已改为持有通用 `Level`，并仅在服务端写入物品 UUID，客户端重建 UI 时只构造界面壳和跳过服务端数据刷新。
- `ConfigInventory`：AE2 1.21.1 的配置背包读写需要 `HolderLookup.Provider`，虚拟物品供应器单元暂用 `GTRegistries.builtinRegistry()` 序列化。后续如 GTM/AE2 提供更明确的上下文来源，需要复核该 registry provider。
- `SavedData`：MC 1.21.1 的 `SavedData` 读写签名新增 `HolderLookup.Provider`，无线能源/CWU 存档先按原有纯数值 NBT 结构补齐新签名。
- `KubeJS/Rhino`：GTM 1.21.1 的 `MachineBuilder` 公开继承 KubeJS `BuilderBase`，因此 GTMM 编译期必须显式加入与 GTM 1.21.1 对齐的 KubeJS/Rhino compileOnly 依赖和 Latvian Maven 仓库。KubeJS 的传递依赖 `com.github.rtyley:animated-gif-lib-for-java` 来自 JitPack，因此仓库侧额外以 `exclusiveContent` 限定 `com.github.rtyley`。这些调整只补齐公开 API/runtime classpath，不改变 Gradle 插件版本。
- `GTItems.attach` 初始化顺序：GTM 1.21.1 当前会在 `CommonProxy.init()` 中先调用 addon `gtInitComplete()`，而 GTM 自身的 `GTItems.init()` 在后续 `RegisterEvent` 阶段。GTMM 物品注册阶段如果调用 `GTItems.attach` 会提前触发 `GTItems.<clinit>`，导致 GTM 材料尚未初始化时创建 `ItemMaterialInfo` 并崩溃。当前 GTMM 使用本地等价 `GTMMItems.attach`，并避免 `CreativeMachines` 直接引用 `GTItems::cellColor`。
- `GTBlocks` 初始化顺序：GTMM 自身的 `RegisterEvent` 可能早于 GTM 的完整内容初始化。`WirelessMultiMachines` 不能在静态注册链中直接引用 `GTBlocks.HIGH_POWER_CASING`，否则会提前触发 `GTBlocks.<clinit>` 并碰到尚未初始化的 GTM 材料键。当前改为通过 `BuiltInRegistries.BLOCK` 延迟解析 `gtceu:high_power_casing`，并用 `Blocks.IRON_BLOCK` 作为异常兜底。
- GTM/GTMM 注册事件顺序：GTM 1.21.1 在 `CommonProxy.onRegister(RegisterEvent)` 中完成 `GTBlocks.init()`、`GTItems.init()`、`GTMachineUtils.init()`、`GTCovers.init()`、`GTMachines.init()`，并在 `modifyRegistries` 中为 `GTRegistries.MACHINES` 挂 `GTMachines::bakeRenderStates`。GTMM 的 `mods.toml` 依赖顺序必须是 `gtceu required AFTER`，否则 GTMM 的机器/多方块注册会先于 GTM 基础内容初始化，提前触发 `GTBlocks` / `GTMachineModels` 静态初始化。当前已将 `GTMMItems` / `GTMMBlocks` 从 `IGTAddon#gtInitComplete()` 移入 GTMM 自己的 `Registries.ITEM` / `Registries.BLOCK` 事件阶段，避免在 GTM addon 回调或事件前阶段提前触发 `GTRegistries.TO_REGISTER`。

## Trait 注册迁移

- `this.attachTrait(...)`：GTM 1.21.1 已改为 `MachineTrait(MetaMachine machine)` 构造时自动调用 `machine.getTraitHolder().attachTrait(this)`，且 `MachineTraitHolder#attachTrait` 不再返回 trait。GTMM 侧已将旧写法 `this.attachTrait(new Trait(...))` 迁移为 `new Trait(this, ...)` 或对应 factory `receiverContainer(this, ...)` / `emitterContainer(this, ...)`。
- 自定义 trait 构造器：`CatalystItemStackHandler`、`CatalystFluidStackHandler`、`InaccessibleInfiniteHandler`、`InaccessibleInfiniteTank`、`ProgrammableCircuitHandler`、无线 CWU/Computation 容器等均改为接收 `MetaMachine` 并传给父类。后续若 GTM 1.21.1 同步回兼容构造器，可复核是否保留当前显式 machine 参数。
- `DemodulationHubMachine.DimensionalRelayNodeBank`：为避免结构重复成型时反复新增 trait，`rebuild` 当前改为在原 trait 实例上刷新容量/损耗统计，而不是重新构造并再次注册。后续如果 GTM 提供 trait 卸载/替换 API，可重新评估是否恢复不可变对象模型。
